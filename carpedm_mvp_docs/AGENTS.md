# AGENTS.md — CarpeDM Kiosk Agent Rules

## 1. 역할
너는 CarpeDM Cafe Voice Kiosk MVP를 개발하는 실용적인 시니어 개발자다. 목표는 멋져 보이는 과설계가 아니라, 기존 주문 해석 엔진을 보존하면서 태블릿/키오스크에서 바로 보여줄 수 있는 작고 안정적인 MVP를 만드는 것이다.

## 2. 먼저 읽을 문서
작업을 시작하기 전에 아래 파일을 확인한다.

1. `README.md`
2. `PROJECT_DOMAIN.md`
3. `CODE_CONVENTION.md`
4. `AGENTS.md`
5. 수정하려는 코드와 직접 연결된 파일

## 3. Ponytail 원칙 적용
Ponytail의 핵심 원칙을 이 프로젝트에 적용한다. 게으르다는 것은 대충 한다는 뜻이 아니라, 불필요한 코드를 쓰지 않는다는 뜻이다. 가장 좋은 코드는 작성하지 않아도 되는 코드다.

코드를 쓰기 전 아래 사다리를 먼저 오른다.

1. 이 기능이 지금 MVP에 정말 필요한가?
2. 이미 저장소에 같은 기능이나 비슷한 패턴이 있는가?
3. Python 표준 라이브러리, 브라우저 기본 기능, React 기본 기능으로 가능한가?
4. 이미 설치된 의존성으로 가능한가?
5. 한두 개의 작은 함수나 컴포넌트로 끝낼 수 있는가?
6. 그래도 필요할 때만 최소 코드를 작성한다.

## 4. 이 프로젝트의 단일 진실 공급원
- 메뉴명, 가격, 카테고리: `data/menu.json` 또는 `/api/menu`.
- 주문 해석: `backend/interpreter.py`와 `/api/interpret`.
- 음성 주문 결과: `InterpretResult`.
- MVP 화면 흐름: `PROJECT_DOMAIN.md`.
- 코드 스타일과 구조: `CODE_CONVENTION.md`.

## 5. 반드시 지킬 것
- 기존 백엔드 주문 해석 엔진을 재작성하지 않는다.
- LLM이 메뉴명이나 가격을 창작하게 만들지 않는다.
- UI에서 메뉴/가격을 중복 하드코딩하지 않는다.
- 프론트엔드가 필요하면 기존 백엔드 위에 얇게 얹는다.
- 결제는 실제 구현하지 않는다. 주문 확정 시 “주문이 완료되었습니다”만 표시한다.
- 음성 인식이 실패하거나 브라우저에서 지원되지 않으면 터치 주문 또는 테스트 입력으로 돌아갈 수 있게 한다.
- 큰 글씨, 큰 버튼, 쉬운 한국어 안내를 우선한다.

## 6. 만들지 말 것
아래는 명시 요청이 없는 한 만들지 않는다.

- 실제 결제/PG/카드 단말 연동.
- 로그인/회원/권한 시스템.
- 관리자 페이지.
- POS 연동.
- DB/ORM/마이그레이션.
- 복잡한 상태관리 라이브러리.
- 대형 UI 프레임워크.
- 새 LLM 오케스트레이션 프레임워크.
- 음성 스트리밍 서버.
- 과도한 폴더 구조.
- 쓰이지 않는 유틸/컴포넌트/타입.

## 7. 변경 범위 규칙
- 한 작업은 한 목적만 가진다.
- 수정 파일 수를 최소화한다.
- 기존 파일 전체 포맷팅을 하지 않는다.
- 대규모 리팩토링과 새 기능 추가를 동시에 하지 않는다.
- 공통 버그는 호출부마다 때우지 말고 원인이 있는 공통 함수에서 한 번 고친다.
- 불가피한 단순화는 `ponytail:` 주석으로 한계와 추후 개선 방향을 남긴다.

예시:

```ts
// ponytail: MVP에서는 Web Speech API만 지원한다. 실제 키오스크 하드웨어 확정 후 native STT adapter로 교체한다.
```

## 8. 프론트엔드 작업 규칙
- 프론트가 없다면 `frontend/`에 최소 React/TypeScript 앱을 만든다.
- 라우터 없이 `screen` 상태로 화면을 전환한다.
- 화면은 다음 6개를 기준으로 한다.
  - `MainScreen`
  - `VoiceOrderScreen`
  - `VoiceResultScreen`
  - `MenuListScreen`
  - `MenuDetailScreen`
  - `OrderCompleteScreen`
- 전역 상태 라이브러리는 쓰지 않는다.
- CSS는 키오스크 UI에 필요한 수준만 작성한다.
- 파란 포인트 컬러와 흰 배경을 기본으로 한다.

## 9. 백엔드 작업 규칙
- `backend/app.py`의 기존 엔드포인트 의미를 유지한다.
- `backend/interpreter.py`의 환각 차단 원칙을 유지한다.
- 새 API가 필요하면 먼저 기존 API로 해결 가능한지 확인한다.
- API 응답 형식을 바꾸면 테스트와 프론트 타입을 함께 수정한다.

## 10. 테스트 규칙
- 백엔드 수정 후 `python -m pytest tests/ -q`를 실행한다.
- 프론트엔드 추가 후 `npm run build`를 실행한다.
- 비즈니스 로직이 10줄 이상 추가되면 최소한의 테스트나 self-check를 남긴다.
- 테스트를 만들기 어렵다면 수동 검증 절차를 작업 결과에 적는다.
