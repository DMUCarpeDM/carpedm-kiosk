## 1. 문제 정의

### 1.1 키오스크는 급증, 고령층은 소외

무인 키오스크 보급은 빠르게 늘고 있다. 국내 키오스크 보급 대수는 2021년 약 21만 대 → 2022년 약 45.4만 대 → 2023년 약 53.7만 대로 2년 새 30만 대 이상 증가했다(한국지능정보사회진흥원). **(신뢰도: 높음)**

그러나 이 전환은 고령층을 구조적으로 배제한다. 핵심 통계는 다음과 같다.

| 지표 | 수치 | 출처 |
|---|---|---|
| 65세 이상 키오스크 주문에 어려움 | **64.2%** | 보건복지부 2020 노인실태조사 |
| 55세 이상 키오스크 이용 불편 경험 | **59.6%** (65–74세 71.5%, 75세+ 76.6%) | 서울디지털재단 2023 디지털역량실태조사 |
| 고령층 키오스크 이용 경험 있음 | **45.8%** (55세 미만 94.1%) | 서울디지털재단 2021 |
| 키오스크 미이용 이유 1위 | '사용법 모름·어려움' **33.8%** | 서울디지털재단 2021 |

**(신뢰도: 높음 — 정부·공공기관 1차 조사)**

### 1.2 "왜 어려운가" — 페인 포인트 3종

한국소비자원 조사 기준, 고령 소비자가 키오스크에서 겪는 어려움은 크게 셋이다. **(신뢰도: 높음)**

1. **조작 복잡성** (51.4%) — 단계가 많고 업종마다 순서가 제각각
2. **뒷사람 눈치 / 시간 압박** (49.0%) — 줄 서 있으면 서두르게 됨
3. **작은 글씨·저시력** (44.1%) — 노안으로 화면 판독 자체가 어려움

학술적으로도 노인의 인지기능(지남력·집중력·기억력)이 키오스크 사용능력과 유의한 상관을 보인다는 연구가 있다(성윤정 외, 대한고령친화산업학회지, 2019). **(신뢰도: 중간 — 표본 10명 소규모 연구)**

### 1.3 지역 근거 — 왜 구로구인가

구로구청 공표 기준 구로구 65세 이상 노인인구는 **85,463명, 전체의 22.2%**로, 서울시 평균(약 18~19%)을 상회한다. **(신뢰도: 중간 — 구로구청 공표값, 기준 시점 확인 권장)** 고령 밀집도가 높은 지역일수록 키오스크 소외의 사회적 비용이 크므로, 리빙랩 실증지로서 타당성이 높다.

### 1.4 기존 대응의 한계 → 음성(VUI)이 본질적 대안인 이유

현재 대응은 대부분 **'고령층 키오스크 교육'** 중심이다. 그러나 "글씨가 작아 화면이 안 보이는데 사용법을 알려줘봐야 소용없다", "알려줘도 기억이 안 난다"는 현장 증언이 반복된다(언론 인터뷰 다수). 즉 **교육은 조작 학습 부담과 인지 부담 자체를 없애지 못한다.**

> **핵심 통찰**: 큰 글씨·고대비 같은 시각 개선은 '저시력' 문제는 풀지만 '조작 복잡성·인지 부담' 문제는 남긴다. **자연어 음성 주문(VUI)** 은 "조작을 학습한다"는 전제 자체를 제거하므로, 페인 포인트 3종 중 ①조작 복잡성과 ②시간 압박을 동시에 겨냥하는 차별적 접근이다. (③저시력은 큰버튼·고대비 UI로 병행 보완)

법·제도적 타이밍도 유리하다. 장애인차별금지법에 따라 키오스크 접근성 제공이 의무화됐고, **2026. 1. 28.부터 기존 설치 키오스크까지 전면 적용** 단계에 들어섰다. 고령·장애인을 위한 접근성 솔루션 수요가 제도적으로 확정된 시점이다. **(신뢰도: 높음 — 법령·보건복지부 자료)**

## 2. 솔루션 컨셉

### 2.1 한 줄 정의

> 라즈베리파이5 기반의 **시니어 특화 AI 음성 가이드 키오스크** — 어르신이 자연어로 말하면(예: "부드러운 거 하나 줘") LLM이 정형 주문으로 변환하고, 음성으로 확인·안내하는 **이동형(탁상형) 독립 모의 주문기**. 기기 하나로 주문이 완결되며, 결제는 직원 호출로 대체한다(POS 연동 없음 — PoC 범위 명확화).
>
> **포지셔닝 — 병설형 시니어 주문 보조 창구**: 기존 키오스크를 대체하지 않고 *옆에 나란히 놓는다*. 매장은 아무것도 바꾸지 않고, 어르신 줄만 사라진다. 기존 키오스크와의 데이터 연동은 없으며(독립 작동), POS 자동 전송은 장기 로드맵으로만 둔다.

### 2.2 핵심 차별점

| 일반 배리어프리 키오스크 | 본 솔루션 |
|---|---|
| 큰 글씨·고대비·TTS로 **기존 터치 UI를 개선** | 터치 학습 부담 자체를 제거하는 **자연어 음성 주문** |
| 사용자가 메뉴 구조를 이해해야 함 | "부드러운 음식", "안 매운 거" 같은 **의도 표현**을 LLM이 해석 |
| 단일 입력(터치) | **멀티모달**: 음성 + 큰버튼 병행 |

### 2.3 대표 사용자 시나리오

```
[어르신 접근] PIR 센서가 감지 → 화면 점등, "안녕하세요, 무엇을 도와드릴까요?" 음성 안내
[발화]        "따뜻한 거 하나랑 부드러운 빵 줘"
[STT]         음성 → 텍스트 변환 (Anker AEC 스피커로 매장 소음 억제)
[LLM]         자연어 → 정형 주문 매핑: {음료: 따뜻한 아메리카노 ×1, 빵: 카스텔라 ×1}
[확인]        화면에 큰 글씨 + 음성으로 "따뜻한 아메리카노 하나, 카스텔라 하나 맞으세요?"
[수정·추가]   "아 참, 콜라도 하나 줘" / "카스텔라는 빼줘" → 장바구니 갱신 후 다시 확인 (멀티턴 대화)
[완료]        "네" 또는 큰 [확인] 버튼 → 주문 완료, 음성+소리 피드백
```

**설계 철칙**: 음성 인식이 실패해도 **큰버튼으로 끝까지 주문을 완료**할 수 있어야 한다(음성은 '대체'가 아니라 '추가' 경로). 타임아웃·재촉 없음.

## 3. 고령층 친화 설계 원칙

설계는 추측이 아니라 **무인정보단말기 접근성 표준 KS X 9211:2022 및 과기부 고시 [별표5]** 를 근거로 한다. **(신뢰도: 높음 — 국가표준·고시)** 본 프로젝트는 인증 대상은 아니지만, 표준을 설계 기준으로 채택하면 "배리어프리 디자인" 주장을 객관적으로 입증할 수 있다.

| 원칙 | 표준 근거 요구사항 | 본 프로젝트 적용 |
|---|---|---|
| 음성 안내 | 음성 출력, 음량 조절(50dB 범위), **다시듣기** 제공 | TTS 안내 + "다시 말해줘" 버튼/명령 |
| 큰 글씨·확대 | 모든 시각 정보 확대·축소 가능 | 기본 대형 폰트 + 글자 크기 토글 |
| 색 대비 | 녹/빨, 파/노 조합 금지·색만으로 구분 금지·흑백 구분 가능 | 고대비 팔레트 + 아이콘/텍스트 중복 표기 |
| 물리 대안 | 모든 시각 정보를 물리 키패드로 선택 가능 | 음성+터치+(선택)물리버튼 |
| 입력 피드백 | 중요 입력 완료 시 소리 정보 제공 | 단계 완료마다 소리+음성 확인 |
| 직관 픽토그램 | 표준 모양의 식별 쉬운 컨트롤 | 표준 아이콘 + 글자 라벨 병기 |
| 시간 압박 완화 | (페인포인트 대응) | 타임아웃 없음, '천천히 모드' |

## 4. 시스템 아키텍처 & 기술 스택

### 4.1 구조

```
[사용자] ──터치/음성/버튼──> [웹 프론트엔드: 접근성 UI]
                                 │            │
                       REST/WS   │            │ TTS·STT·LLM
                                 ▼            ▼
                          [로컬 서버:FastAPI]  [AI 모듈]
                                 │
                              GPIO │
                                 ▼
                          [하드웨어: 스피커·PIR센서·(키패드)]
        └──────── 라즈베리파이5 키오스크 ────────┘
```

### 4.2 기술 선택과 트레이드오프

| 영역 | 선택 | 대안 | 트레이드오프 / 신뢰도 |
|---|---|---|---|
| 프론트엔드 | React | 바닐라 JS | React는 접근성 모드 토글·상태관리 용이. 바닐라는 의존성↓·파이 성능 유리. 단일 화면이라 React로 충분 |
| 음성 인식(STT) | **확정: 클라우드 STT 우선 + 오프라인 폴백**(faster-whisper, 파이 로컬) | 온디바이스 단독 | 클라우드가 빠르고 정확(파이 로컬 수 초 vs 클라우드 ~1초 사례). 폴백은 네트워크 장애 대비. 고령층 발화 정확도는 여전히 최대 변수(실측 필요) |
| 자연어 처리(LLM) | **확정: Claude API 멀티턴 주문 대화** — 장바구니 상태+대화 맥락을 함께 전달, 자연어→정형 주문(JSON) | 규칙 기반 매핑 | **지원 명령 5종으로 범위 한정**: 추가 / 수량 변경 / 항목 제거 / 전체 취소 / 확인. API 키는 백엔드 환경변수로만 보관(프론트 노출 금지). 비용은 호출당 소액 예상(실측·한도 설정) |
| 음성 출력(TTS) | **확정: 클라우드 TTS 우선 + 브라우저/로컬 폴백** | — | 자연스러운 한국어 음성이 고령층 신뢰에 중요. 폴백 시 음질 저하는 감수 |
| 백엔드 | FastAPI (Python) | Flask | GPIO·센서 제어 라이브러리 풍부 |
| 소음 대응 | Anker PowerConf S3 (하드웨어 AEC) | 소프트웨어 노이즈 제거 | 매장 소음 환경 대응. (스피커폰 사양은 구매 전 재확인 권장) |

### 4.3 1순위 기술 리스크 — 고령층 발화 인식률

매장 소음 + 방언·느린 발화·말끝 흐림 조건에서 STT 정확도가 떨어질 수 있다. **이것이 프로젝트 성패의 핵심 변수다(신뢰도: 낮음, 반드시 실측).**

대응 설계:
- **멀티모달 완주 보장**: 음성 실패 시 큰버튼으로 동일 주문 완료
- **재확인 루프**: 인식 결과를 항상 음성+화면으로 되묻고 확정
- **발화 데이터셋 수집**: 실증에서 모은 고령층 발화로 인식 실패 패턴 분석 → 고도화 근거 + 학술 자산화

## 5. 실증 계획 (PoC) — 경로당·주민센터 전환

> **변경**: 복지관 멘토링/실증을 제외하는 대신, **경로당·주민센터 등 지역시설을 섭외**하여 실사용자 실증을 수행한다. 리빙랩의 핵심(실사용자 현장 검증)은 그대로 유지된다.

### 5.1 섭외 전략

- 구로구·금천구 **경로당, 동주민센터 어르신 프로그램, 노인복지관(멘토링 없이 장소·대상 협조만)** 우선 접촉
- 학교(RISE 사업단·지도교수) 명의 공문/협조 요청으로 신뢰도 확보
- 1차 실패 대비 **복수 후보지 동시 접촉** (섭외 실패가 가장 큰 일정 리스크)
- 섭외 착수를 5월 2~3주로 **앞당김** (경로당은 리드타임이 길다)

### 5.2 실증 설계

- **대상/인원**: 65세 이상 어르신 (최소 8~12명 권장 — 사용성 평가는 5명만으로도 주요 문제의 다수 발견이 가능하다는 것이 통설이나, 정량 지표 신뢰도를 위해 8명 이상 권장)
- **과업 시나리오**: 표준 주문 과업 2~3종(예: 음료 1종 주문 / 음료+음식 2종 / 주문 수정)
- **측정 지표 — 1순위 = 음성 매핑 정확도 (인터뷰 확정)**
  - 정의: 발화 → 시스템이 산출한 정형 주문(JSON)이 정답 주문과 **완전 일치**(항목·수량·옵션)한 비율
  - **2단 분리 측정**: ① STT 전사 정확도(참고) ② 텍스트→주문 매핑 정확도(핵심). 분리해야 오류 원인이 '귀(STT)'인지 '머리(LLM)'인지 알 수 있다
  - 테스트셋: 단일 발화 70문장 + 멀티턴 시나리오 30개(제안)로 자동 측정
  - **제안 목표치(팀 설정값, 실측 후 조정)**: 테스트셋(텍스트) ≥90% / 현장 1차 발화(음성 포함) ≥75% / 되묻기 복구 포함 주문 완료율 ≥95%
  - 전체 지표:

| 지표 | 정의 |
|---|---|
| 과업 완료율 | 도움 없이 주문을 끝낸 비율 |
| 평균 소요시간 | 시작~완료 (비교: 기존 터치 키오스크 대비) |
| 오류·취소 횟수 | 잘못 선택/되돌리기 빈도 |
| 도움 요청 횟수 | 진행자에게 물은 횟수 |
| 음성 1차 인식 성공률 | 첫 발화로 의도 인식된 비율 |
| SUS | System Usability Scale 표준 설문(0–100) |
| 주관 만족도 | 5점 척도 + 자유 코멘트 |

- **비교 설계**: 가능하면 '기존 터치 방식' vs '음성 방식' 과업 시간·완료율을 비교해 효과를 정량 입증

### 5.3 윤리·개인정보

- 음성 녹음·발화 데이터 수집 시 **사전 동의서** 필수(목적·보관기간·파기·익명화 명시)
- 신청서의 개인정보 처리 방침과 정합되게 관리, 수집 데이터는 연구 목적 외 사용 금지

## 9. 평가(성과발표) 대비 강점

- **문제 정의의 근거력**: 정부·공공 통계 + 지역 데이터 + 학술 근거로 무장 (위 1장)
- **차별성**: '교육'이나 '큰 글씨'에 머무는 기존 대응과 달리 **자연어 음성으로 조작 부담 자체를 제거**
- **표준 정합성**: KS X 9211 기반 설계로 '배리어프리'를 객관 입증
- **리빙랩 충실성**: 경로당·주민센터 실사용자 실증 + 정량 지표(SUS 등)
- **확장성**: 병설형이라 기존 인프라 무변경 보급(매장·공공 어디든 '한 대 추가'), POS 연동 시 실주문 자동 전송(장기 로드맵), 구청 민원실·보건소 이식, 시각장애인·다문화 확장
