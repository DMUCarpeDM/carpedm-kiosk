# 요구사항 정의서 (SRS v1)
### CarpeDM — 고령층 AI 음성 가이드 키오스크 (병설형 시니어 주문 보조 창구)

> 문서 목적: 개발 범위를 기능 단위로 확정하고, 각 기능에 **검증 가능한 수용 기준**을 부여한다. 수용 기준은 그대로 Claude Code `/goal` 조건·pytest 케이스가 된다.
> 관련 문서: ① 개정기획서 v2 ② 차별화전략·리서치의제 ③ 데이터근거 자료집 ④ IA·유저플로우
> 우선순위: **M**(Must, 데모 필수) / **S**(Should, 실증 전 필요) / **C**(Could, 여유 시)
> ※ 수치 목표는 팀 설정값(제안)이며 실측 후 조정 가능. ⚠는 실측·검증 필요 항목.

---

## 0. 용어
| 용어 | 정의 |
|---|---|
| 발화 | 사용자가 말하거나 입력한 자연어 문장 |
| 정형 주문 | `{items:[{id, qty}], ...}` 형태의 JSON. 메뉴는 **id로만** 참조 |
| 매핑 정확도 | 발화 → 정형 주문이 정답과 완전 일치(항목·수량·옵션)한 비율 |
| 명령 5종 | 추가 / 수량 변경 / 항목 제거 / 전체 취소 / 확인 |
| 폴백 | 클라우드(STT·TTS·LLM) 불가 시 로컬 대체 동작 |

## 1. 시스템 구성 & 담당(제안)
| 모듈 | 내용 | 담당 |
|---|---|---|
| FE | 키오스크 웹 UI (React, 풀스크린) | 김주현·이병윤 |

| DATA | 테스트셋·측정 스크립트·표현 사전 | 전원(취합 팀장) |

## 2. 기능 요구사항 (FR)

### 2.1 음성 입출력 (V)
| ID | 우선 | 요구사항 | 수용 기준 (검증 방법) |
|---|---|---|---|
| FR-V1 | M | 마이크 버튼으로 음성 입력 시작, 인식 텍스트 실시간 표시 (클라우드 STT) | 조용한 환경 표준 발화 10건이 화면에 전사되어 표시됨. STT 오류 시 에러 화면 대신 "다시 말하기/직접 입력" 안내가 뜸 |
| FR-V2 | S | 네트워크 단절 시 로컬 STT(faster-whisper) 자동 전환 | 네트워크 차단 상태에서 음성 주문 1건 완료 가능(지연 허용 ⚠실측). 전환 이벤트가 로그에 기록됨 |
| FR-V3 | M | 모든 안내를 TTS로 출력 (클라우드 우선, 브라우저/로컬 폴백). 말속도 기본 '느리게' | 8개 화면 전환마다 음성 안내가 출력됨. 폴백 모드에서도 음성 출력됨 |
| FR-V4 | M | '다시 듣기' — 직전 안내 멘트 재생 (KS X 9211) | 모든 화면에서 다시 듣기 버튼이 직전 멘트를 재생함 |
| FR-V5 | M | **음성 외 입력은 큰 그림 버튼(터치)으로만 제공한다. 사용자용 키보드/타이핑 입력은 두지 않는다** (고령층 문자 입력 부담 = 해결 대상 문제이므로 폴백으로도 부적합) | UI 전체에 텍스트 입력 필드 0건. 음성 실패 시 안전망은 메뉴 화면의 터치 경로가 담당함 |

### 2.2 자연어 해석 (N) — 핵심 차별 모듈
| ID | 우선 | 요구사항 | 수용 기준 |
|---|---|---|---|
| FR-N1 | M | 단일 발화 → 정형 주문 해석. 은유·간접 표현("검은 탄산 물") 매핑. **LLM은 메뉴 id만 반환**, 이름·가격은 시스템이 menu.json에서 조회 | 단일 발화 테스트셋 70건 매핑 정확도 ≥90% (자동 측정 스크립트 기준) |
| FR-N2 | M | 멀티턴 수정 — 장바구니 상태+직전 맥락을 포함해 명령 5종 해석 ("콜라 빼줘", "하나 더") | 멀티턴 시나리오 30건 정확도 ≥85%, 전체 100건 ≥90% 동시 충족 |
| FR-N3 | M | 애매하면 되묻기 — 후보 2~4개를 질문+버튼으로 제시. 임의 확정 금지 | "부드러운 거" 류 모호 발화 10건에서 임의 확정 0건, 전부 되묻기 동작 |
| FR-N4 | M | 메뉴 외 요청 시 가까운 대안 제안 ("그건 없어요, ○○은 어떠세요?") | 메뉴 외 발화 5건에서 '없음' 안내+대안 제시, 환각 메뉴 생성 0건 |
| FR-N5 | S | 속성 분해 매핑 — 온도/맛/사이즈 속성 조합으로 긴 메뉴명 대응 | 속성 조합 발화 10건("달달한 거 큰 걸로 차갑게") 정확도 ≥80% ⚠ |
| FR-N6 | C | 쉬운 이름 자동 생성 — 메뉴 CSV 입력 → 쉬운 설명 초안 생성 → 검수 파일 출력 | 30개 메뉴 CSV 입력 시 쉬운 이름 초안 30건 생성, 검수용 파일 저장 |
| FR-N7 | S | **추천 — "뭐가 맛있어요?", "따뜻한 거 추천해줘"** 등에 메뉴 1~3개 제안(추천 이유 1문장). 조건(온도·맛·분류) 반영. 추천 id도 메뉴 내 존재만(환각 차단) | 추천 요청 6건에서 action=recommend, suggestions 전부 메뉴 내 id, 조건(온도/맛) 위반 0건 |

### 2.3 화면·접근성 (U)
| ID | 우선 | 요구사항 | 수용 기준 |
|---|---|---|---|
| FR-U1 | M | 8개 화면 흐름 = IA 다이어그램과 일치 (대기→시작→듣기/메뉴→해석→되묻기→확인→수정→완료·호출) | 화면 전환 경로가 IA와 1:1 대응(수동 점검 체크리스트 통과) |
| FR-U2 | M | 메뉴 우리말 우선 표기 + 원어 병기, 사진/픽토그램 병행 | 모든 메뉴 카드에 쉬운 이름(대)+원어(소) 표기 |
| FR-U3 | M | 글자 크게 토글(≥1.2×), 기본부터 대형 폰트 | 토글 시 전 화면 글자 확대, 새로고침 전까지 유지 |
| FR-U4 | M | 큰 터치 영역 — 주요 버튼 높이 ≥72px, 간격 ≥12px | 전 화면 버튼 측정 통과 (W3C 48px 기준 상회) |
| FR-U5 | M | 고대비·파랑 회피 팔레트, 색+글+아이콘 3중 표기 | 본문 대비 7:1 목표(WebAIM 검사 ⚠), 색만으로 구분되는 정보 0건 |
| FR-U6 | M | 타임아웃·재촉 없음. '처음으로' 상시 제공 | 5분 방치에도 화면 유지, 모든 화면에 처음으로 버튼 |
| FR-U7 | M | 주문 확인 화면 — 내역 큰 글씨 + 음성 재확인 후에만 확정 | 확인 없이 주문 완료되는 경로 0건 |

### 2.4 주문·하드웨어 (O/H)
| ID | 우선 | 요구사항 | 수용 기준 |
|---|---|---|---|
| FR-O1 | M | 장바구니 상태 관리 (담기·수량·제거·전체취소), 음성/터치 양 경로 공유 | 양 경로 혼합 사용 시나리오 1건(음성 담기→터치 수정) 정상 동작 |
| FR-O2 | M | 주문 완료 → 직원 호출 (알림음 + 주문 내역 화면 유지) | 완료 시 알림음 1회+내역 표시. 실결제 요소 없음 |
| FR-H1 | S | PIR 감지 → 대기 화면 해제 + 인사 음성 | 1m 내 접근 시 5초 내 인사 출력 ⚠실측 |
| FR-H2 | S | 부팅 → 자동 키오스크 모드(크롬 풀스크린) 진입 | 전원 인가 후 무조작 60초 내 시작 화면 도달 ⚠ |

### 2.5 데이터·백엔드 (D/B)
| ID | 우선 | 요구사항 | 수용 기준 |
|---|---|---|---|
| FR-D1 | M | 발화 로깅 — {시각, 발화, STT결과, LLM출력, 최종주문, 폴백여부}를 JSONL 저장. **개인 식별정보 없음** | 주문 10건 후 로그 10행, 식별 필드 부재(스키마 검사) |
| FR-D2 | M | 정확도 측정 스크립트 — 테스트셋 실행 → accuracy.md 리포트 자동 생성 (전체/단일/멀티턴/속성별) | 단일 명령 실행으로 리포트 생성, CI에서 pytest 통과 |
| FR-D3 | S | 표현 사전 외부화 — 코드가 아닌 데이터 파일(yaml/json)로 관리, 현장 수집분으로 교체 가능 | 사전 파일 교체만으로 재배포 없이 매핑 반영 |
| FR-B1 | M | LLM API 프록시 — 키는 서버 .env에만 존재 | 프론트 번들 전체 grep에서 키 문자열 0건, /api/interpret 경유 확인 |
| FR-B2 | M | menu.json 단일 소스 — 가격·이름·id의 유일한 출처 | LLM 응답의 id가 menu.json에 없으면 거부 처리(테스트 3건) |
| FR-D4 | M | **실증 모드** — 숨김 제스처(예: 로고 5회 탭)로 참여자 번호(P-xx) 입력 → 이후 로그에 participant 태그 (임의 번호라 P-6 위배 아님). 기록지와 자동 로그를 연결하는 장치 | 태깅 상태에서 주문 2건 수행 시 로그 2행에 participant 값 기록 |
| FR-D5 | S | **결과 자동 집계** — 발화 로그(+기록지 CSV)에서 과업 완료율·평균 소요·되묻기 분포·폴백률 산출 | `report.py` 1회 실행으로 reports/field_summary.md 생성 |

## 4. 정책 정의 (P)
- **P-1** 명령 5종 외 발화는 실행하지 않고 되묻는다 (임의 행동 금지)
- **P-2** 모든 주문은 재확인("맞으세요?") 후에만 확정된다
- **P-3** 타임아웃·재촉 멘트 금지 (페인포인트 '눈치 49%' 대응)
- **P-4** 음성 인식 연속 2회 실패 시 버튼 경로를 음성으로 적극 안내
- **P-5** LLM은 id만 산출, 가격·메뉴명은 시스템이 조회 (환각 차단)
- **P-6** 로그에 이름·전화번호 등 식별 정보를 절대 수집하지 않는다

## 5. 범위 제외 (Out of Scope)
실결제·POS 연동(장기 로드맵으로만 기재) / 회원·적립 / 다국어 / 원격 관리 콘솔 / 안면 인식

## 6. `/goal` 매핑 예시 (Claude Code 투입용)
- `/goal FR-N1·N2: 해석 모듈과 테스트셋 100건이 구현되고, python measure.py 실행 시 accuracy.md가 생성되며 전체 ≥90%·멀티턴 ≥85%를 충족하고 pytest가 전부 통과한다. 테스트 실행 출력을 보여줄 것`
- `/goal FR-B1·B2·D1: FastAPI 백엔드가 /api/interpret를 제공하고, 프론트 grep에 API 키가 없으며, 주문 10건 시뮬레이션 후 로그 JSONL 10행이 생성됨을 테스트로 증명한다`
- `/goal FR-U1~U7: 8개 화면이 IA와 일치하고 접근성 체크리스트 스크립트(버튼 크기·대비)가 통과한다. 스크린샷 또는 검사 출력 제시`
